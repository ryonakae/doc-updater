# doc-updater workflow

`git diff --cached` に含まれる変更が git 管理ドキュメントに与える影響を判断し、必要な更新だけを行うための shared workflow。メインエージェントと、独立したコンテキストに切り出されたサブエージェントのどちらからでも実行できる。

## 前提

- 対象は staged change だけ
- シンボリックリンクは直接編集しない
- 差分は最小限に留める
- コミットや push は行わない
- ドキュメントを更新したら `git add` でステージングに追加する
- 呼び出し元から staged file list と rename/delete context が渡されている場合は、それを優先して使う
- サブエージェントとして実行された場合は、最後に呼び出し元へ更新内容を簡潔に返す

## スキップしてよい条件

- 呼び出し元が「このセッションで既にドキュメント更新済み」と明示している
- 呼び出し元が「更新不要と判断済み」と明示している
- ドキュメントファイルがすでに staged されていて、今回のコード変更で追加更新が不要と判断できる
- 変更が docs だけで、ループ防止のため追加更新が不要

スキップする場合は「ドキュメント更新は不要です」とだけ返して終了する。

## 1. ドキュメント構造の検出

ルート直下のエージェント指示ファイルは、実ファイルだけを対象にする。

```bash
find . -maxdepth 1 \( -name "AGENTS.md" -o -name "CLAUDE.md" -o -name "GEMINI.md" \) ! -type l
```

- 出力された実ファイルをルート編集対象候補にする
- `README.md` は人間向けドキュメントとして別に確認する
- `README.md` が symlink の場合はリンク先の実ファイルだけを編集する

分割ドキュメント構造がある場合は追加で確認する。

```bash
find ./docs/agents ./docs/agent -name "*.md" -type f 2>/dev/null || true
```

- `docs/agents/` と `docs/agent/` の両方がある場合は、ルートからリンクされている側を優先する
- トピックは「ファイル名」「H1 見出し」「ルートからのリンク文言」の 3 点で把握する

## 2. staged 変更の把握

変更確認には次を使う。

```bash
git diff --cached --name-only --diff-filter=ACMRD
git diff --cached
git diff --cached --diff-filter=R --name-status
git diff --cached --diff-filter=D --name-only
```

- リネームされたファイルは旧パスと新パスの両方を把握する
- 削除されたファイルはドキュメント内に参照が残っていないか確認する

## 3. 更新要否の判定

以下は更新する。

- 機能追加、API 変更、振る舞い変更
- エージェント、スキル、フック、CLI workflow の変更
- 依存関係ファイルの変更
- ビルド、テスト、実行コマンドの変更
- 設定ファイルの変更
- ファイルやディレクトリのリネーム、削除
- README や AGENTS 系に書かれている手順、構成、制約が古くなる変更

以下は更新不要。

- 変数名変更、関数抽出、内部リファクタリングだけで仕様が変わらない
- テストやコメントだけの変更
- ドキュメントファイルだけの変更

迷ったら更新する側に倒す。

## 4. 更新先の決定

分割構造がない場合はルートファイルに書く。

分割構造がある場合は次で振り分ける。

- トピック固有の詳細は該当する分割ファイル
- プロジェクト全体に関わる変更はルートファイル
- 分割ファイルの追加、削除、リネームがある場合はルートのポインタリンクも更新
- 新トピックは既存の近い分割ファイルへの追記を優先し、難しければ新規ファイルを作ってルートにポインタを追加

## 5. 更新時の原則

- 既存の構造とトーンを維持する
- コードを長く複製せず、ファイルパスやコマンドで示す
- 存在しないパス参照は更新または削除する
- エージェント指示ファイルは WHY / WHAT / HOW を優先する
- 自明な説明や重複した指示は増やさない
- symlink を壊さず、実ファイルだけを編集する

## 6. 検証

更新後は次を確認する。

- ドキュメント内のファイルパス参照が実在する
- エージェント指示ファイルを更新した場合は 300 行未満に収まっている
- symlink を直接編集していない

## 7. 出力

最後に次を簡潔に報告する。

- 編集対象として選択したファイル
- 更新したファイル名と変更概要
- 更新をスキップしたファイルとその理由
- エージェント指示ファイルを更新した場合は現在の行数
